# Draft Guidelines

Use these rules when writing reply drafts in Phase 4. These apply to every draft, no exceptions.

---

## The Non-Negotiables

1. **Never send.** Create a draft. The user reviews and sends.
2. **No filler.** Cut every phrase that adds length without adding meaning.
3. **Match length to the ask.** A one-line question gets a one-line answer.
4. **Use `[...]` for unknowns.** Never invent facts. Placeholder what you don't know.

---

## Tone Matching

Read the incoming email before writing. Match its register:

| Incoming tone | Your draft tone |
|---------------|----------------|
| Formal (HR, legal, executive) | Professional, complete sentences, no contractions |
| Collegial (coworker, manager) | Conversational, direct, first-name basis fine |
| Casual (friend, peer) | Relaxed, matching their energy |
| Terse (one-liner) | One-liner back |

Do not default to corporate speak when the original email is casual. Do not match nervousness or over-apologizing.

---

## Structure

For most emails, this three-part structure works:

1. **Acknowledge** — one sentence max, only if needed (e.g., "Thanks for sending this over.")
2. **Answer / Act** — directly address the ask. This is the whole draft.
3. **Close** — one short line if needed. Often nothing.

Skip the acknowledge if the email is a task or request with no pleasantries. Skip the close if the body already lands cleanly.

---

## Filler Phrases — Never Use These

These phrases add zero value and make the user look like a template:

- "I hope this email finds you well"
- "Please don't hesitate to reach out"
- "Feel free to let me know if you have any questions"
- "I wanted to follow up on..."
- "Thank you so much for your email"
- "I look forward to hearing from you"
- "Please advise"
- "As per my last email"
- "Best regards" / "Kind regards" / "Warm regards" (use "Best," or nothing)

---

## Using Calendar Context

When the email involves scheduling:
- Propose **specific slots** from the calendar — not "sometime next week"
- Offer **2–3 options** with day, date, and time: "Tuesday May 12 at 2pm or Thursday May 14 at 10am"
- Note the timezone if it might differ from the recipient
- If the calendar shows nothing free, say so and ask what works for them instead

---

## Using Notes Context

When context files are relevant:
- Pull **specific facts**: project names, decisions, numbers, commitments
- Don't quote the notes verbatim — weave in the relevant detail naturally
- If the notes contradict the email's assumption, flag it with `[Note: ...]` for the user to review

---

## Placeholders

Use `[...]` wherever the user needs to fill in:
- Specific numbers or amounts you don't have: `[amount]`
- Names of people you don't know: `[team name]`
- Decisions only the user can make: `[confirm yes/no]`
- Dates the user needs to choose: `[your preferred date]`

Make placeholders obvious — the user should be able to scan the draft and spot everything they need to fill in.

---

## When to Skip a Draft

Mark as `draft_skipped` if:
- The email is completely ambiguous ("Hey, let me know what you think" with no context given)
- The action required is a real-world action (pay an invoice, sign a document) with no reply needed
- You have no useful information to add beyond "okay" — in which case the user can reply themselves in 2 seconds

Always explain why in the triage report.

---

## Length Reference

| Email type | Draft length |
|------------|-------------|
| Yes/no question | 1 sentence |
| Scheduling request | 2–3 sentences (availability + options) |
| Information request | 1–3 sentences per question asked |
| Document review request | 2–4 sentences (acknowledge + timeline or questions) |
| Job/interview scheduling | 3–5 sentences (interest + availability + next step) |
| Complex multi-part ask | Up to 2 short paragraphs — never more |
